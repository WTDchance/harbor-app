# HIPAA audit log coverage — `app/api/ehr/*` and `app/api/admin/*`

_Snapshot 2026-04-26 (Wave 36 + Tier-1 push)._

The matrix below was generated by grepping each route file under
`app/api/ehr` and `app/api/admin` for any of:

* `auditEhrAccess` (the standard EHR audit helper, `lib/aws/ehr/audit.ts`)
* `auditSystemEvent`
* `audit_logs` direct INSERT

A route is "AUDIT" if at least one such call exists in the file. A route is
"MISS" if none does. PHI determination is by reading the route source, not
the path alone — a few "MISS" rows have legitimate reasons (no PHI touched).

## Suggested PR shape

Group fixes into commits by domain so review is small:

1. `feat(audit): add audit calls to EHR appointment/session and homework/[id] routes`
2. `feat(audit): add audit calls to messages/[id] and mood-logs`
3. `feat(audit): add admin.patient.list to admin/patients/route.ts`
4. `chore(audit): defer the rest after a NEEDS-REVIEW pass on listed routes`

Each fix should be ~3 lines: import `auditEhrAccess`, call after the success
branch with the appropriate action enum (extend `EhrAuditAction` if needed).

## Why we didn't land the fixes tonight

A correct audit call needs the `ApiAuthContext` already in scope (which it
is for these routes), the right action enum string, and a `target_*` plus
optional metadata. Five EHR + 3 admin = eight 3-line edits, but each
needs a quick read of the surrounding route to confirm the right action
name. That's a 30-minute careful pass, not a 2-AM bulk-sed. Filed as an
early-morning follow-up so the user can review the action enum names
before they're frozen into the audit-log schema for posterity.
